I. Introducción y Marco Arquitectónico
1. Descripción General y Alcance del Proyecto
1.1. Propósito del Sistema y Alcance
El sistema "Kalyo TKD - Poomsaes Scoring" se define como una aplicación ejecutable (.exe) diseñada para gestionar y registrar las puntuaciones de las competencias de Poomsae de Taekwondo, alineándose con las normativas de la World Taekwondo (WT). Su principal alcance es proporcionar una interfaz de registrador eficiente y una visualización pública sincronizada para eventos de rondas y pirámides.
1.2. Tecnologías Core (Stack Técnico)
El marco tecnológico propuesto está sólidamente orientado al desarrollo de aplicaciones de escritorio modernas con foco en la portabilidad y el rendimiento local:
Componente
	Tecnología
	Razón / Viabilidad Técnica
	Fuente
	Tipo de Aplicación
	Ejecutable de Escritorio
	Garantiza la operación sin dependencia de conexión a internet o navegador.
	

	Frontend/Lógica
	Type Script y Vanilla JS
	Proporciona tipado estático (Type Script) para robustez del código y alto rendimiento (Vanilla JS) en la interfaz del registrador.
	

	Estructura/Packaging
	Tauri
	Actúa como wrapper o framework para el desarrollo de aplicaciones de escritorio ligeras y multiplataforma, facilitando la distribución como un .exe.
	

	Persistencia de Datos
	SQLite
	Base de datos local ligera. Crítica para garantizar la persistencia de datos ante "cierres inesperados" o recesos, permitiendo "retomar la competencia sin pérdida de información".
	

	Análisis Full Stack Senior (Viabilidad): La elección de Tauri con Type Script y SQLite es óptima para un sistema de puntuación en eventos, ya que minimiza la latencia (operando de forma nativa) y garantiza la integridad de la información crítica de la competencia de manera local.
2. Estándares de Diseño y Experiencia de Usuario (UI/UX)
La Experiencia de Usuario (UX) de este sistema debe priorizar la eficiencia del flujo de trabajo del registrador y la claridad de la información, dada la naturaleza de alta presión del entorno de competencia.
2.1. Estética General y Estilo Visual (UI)
El diseño sigue una estética minimalista.
* Fondo Principal: Color blanco.
* Acentos y Contraste: Uso de bloques y líneas sutiles en tonos rojo y azul.
* Aplicación de Color (UX Clave): El uso de bloques de color primario es funcional para la navegación inicial:
   * Azul: Asociado a la creación y gestión de elementos nuevos o primarios (ej. "Crear Nuevo Evento").
   * Rojo: Asociado a la continuidad, estado o acceso a elementos existentes/en curso (ej. "Continuar Evento").
2.2. Principios de Accesibilidad y Eficiencia del Flujo (UX)
Para asegurar un flujo de trabajo eficiente, se aplicarán los siguientes principios de diseño, derivados de los requisitos del sistema:
1. Código de Color Funcional: La diferenciación Rojo/Azul no solo se aplica a la navegación inicial, sino que se extiende a la interfaz de puntuación de pirámides para identificar rápidamente a los competidores (Competidor Azul vs. Competidor Rojo).
2. Sincronización Crítica: Se requiere una sincronización en tiempo real entre la interfaz principal del registrador (Página 4.0) y la Pantalla Pública (PDI). Esto es vital para la credibilidad y transparencia del evento, donde el ingreso de puntajes se reflejará "de forma inmediata y automática".
3. Animación y Retroalimentación Temporal: El inicio de la aplicación incorpora una animación sutil de 2 a 3 segundos para feedback visual. La visualización de resultados clave, como el anuncio del ganador en pirámides, se mantiene estática por 7 segundos para asegurar la comprensión por parte del público y los atletas.
4. Generación Automatizada de Títulos: Para reducir errores de ingreso de datos por parte del registrador, el título de la categoría se autogenerará con base en las selecciones de Disciplina, Edad, Nivel, etc..
5. Persistencia Robusta: El uso de SQLite garantiza que, en caso de fallo, la data esencial (eventos, categorías y puntajes parciales) esté protegida.

10. Configuración de Rondas (Bloque Específico)
Este bloque se habilita si el sistema es Cut-off (Rondas).
10.1. Registro de Competidores (Mejora UX: Pegado de Datos)
El enfoque principal es la eficiencia en la entrada de grandes volúmenes de datos.
1. Número de Competidores (Q): Campo numérico.
2. Tabla de Entrada Dinámica: Se presenta una tabla con las columnas Nombre y Delegación.
3. Funcionalidad de Pegado (Paste):
   * UX/UI: El área de la tabla de competidores debe tener un mensaje claro que indique: "Pegar datos aquí (formato: Nombre; Delegación)".
   * Viabilidad Técnica (Dev): El front-end (Vanilla JS/Type Script) debe implementar una función robusta de parsing que detecte la estructura tabular copiada desde una hoja de cálculo (ej. Excel, Google Sheets), asumiendo que los datos están segregados por tabulación, coma o punto y coma, y que las columnas coinciden con el orden Nombre; Delegación. Esto poblará la tabla automáticamente, estableciendo el orden de competencia inicial según el orden de pegado.
10.2. Lógica de Sistema de Rondas (Automatizada)
La aplicación debe calcular automáticamente el flujo de rondas en el back-end (Type Script) y reflejarlo en la interfaz:
* Q ≤ 8: Ronda Única (Final).
* Q > 8: Ronda Clasificatoria (Todos ejecutan) → Ronda Final (8 mejores).

11. Configuración de Pirámides (Bloque Específico - 1 vs 1)
Este bloque se activa si el sistema es Elimination Tournament (Pirámide). La complejidad radica en la gestión de las llaves y los Bye (VAI).
11.1. Gestión de Competidores (Pegado y Numeración)
Se utiliza la misma funcionalidad de Pegado de Datos (Nombre; Delegación) que en la sección de Rondas (Sección 10.1). El orden de pegado establece la numeración inicial de los competidores (Posición 1, 2, 3, etc.) en la llave.
11.2. Visualización y Manipulación de la Pirámide (Mejora UX: Scroll)
El UX se centra en una vista interactiva de la llave de eliminación:
* Wireframe Conceptual (Vista Dinámica): El área principal del bloque se transformará en una visualización de llave de árbol horizontal (Pirámide).
* Funcionalidad Scroll: La visualización debe ser desplazable (scrollable) horizontalmente (para ver las diferentes Fases: 16avos, Octavos, Cuartos, Semi, Final) y verticalmente (para ver todos los encuentros).
* Manipulación (UX/UI): Antes del inicio de la puntuación, el registrador debe poder arrastrar y soltar competidores dentro de la llave, si es necesario, para ajustar la Llave de Eliminación "a su antojo", aunque la recomendación es seguir la lógica automática de sorteo.
11.3. Lógica Automática de Confección de Llaves y VAI
El sistema debe calcular automáticamente la estructura de la llave basándose en el Número BASE (potencia de 2 superior inmediata) y el Número de VAI (Bye), como se define en la lógica del sistema.
* Cálculo (Dev): El sistema identifica la Fase de la Llave inicial, el Número BASE (ej., 8 para 7 competidores) y los Competidores que pasan por VAI (Número BASE - Competidores).
* Manejo de VAI (Mejora UX): La interfaz visual de la pirámide (el scrollable tree) debe mostrar a los competidores que avanzan por VAI (Bye) avanzando automáticamente a la siguiente ronda.
   * Constraint Fulfillment: Se cumple la restricción de no mostrar encuentros N/A contra N/A, ya que el competidor con VAI simplemente aparece en la fase siguiente, sin crear un encuentro de placeholder en la fase inicial.
11.4. Gestión de Encuentros y Flujo de Avance
Una vez generada la pirámide:
* Selección de Encuentro: El registrador selecciona la Fase de la Llave y el Número de Encuentro.
* Población Automática (Dev): Los campos Competidor Azul y Competidor Rojo se pueblan automáticamente con los nombres (Delegación y Nombre) correspondientes a ese encuentro, utilizando los ganadores de rondas anteriores (o los competidores iniciales).
* Gestión de Ausencia (VAI por Retiro): El botón "Uno de los competidores gana por VAI (ausencia)" se mantiene. Si se presiona, la lógica interna registra al oponente como ganador y el botón "Iniciar Puntuación" cambia a "Guardar Ronda", sin requerir calificaciones.
* Asignación Automática de Podio: El sistema debe tener precargada la lógica de asignación de 3er y 4to lugar (Perdedor de Semifinal vs. Campeón, Perdedor de Semifinal vs. Subcampeón).

VII. Lógica de Implementación (Dev) y Algoritmos de Puntuación
Esta sección define el núcleo matemático y algorítmico del sistema, vital para la fiabilidad de la aplicación, independientemente de si el flujo es por Rondas o por Pirámides.
15. Algoritmo de Cálculo de Puntaje Total (Regla WT)
El cálculo del puntaje total de un competidor se basa en la suma de los promedios de la Calificación de Técnica ($\text{Prom}{T}$) y la Calificación de Presentación ($\text{Prom}{P}$).
15.1. Definición de Variables
* $N$: Número total de jueces (3, 5 o 7).
* $T_j$: Calificación de Técnica otorgada por el juez $j$.
* $P_j$: Calificación de Presentación otorgada por el juez $j$.
15.2. Regla de Descarte de Puntajes (Descarte WT)
El sistema debe determinar el conjunto de calificaciones válidas ($N'$) antes de calcular el promedio:
Condición (N = Jueces)
	Regla de Descarte
	Divisor ($N'$)
	$N=3$
	Se utilizan todos los valores.
	$N' = 3$.
	$N=5$ o $N=7$
	Se elimina el valor máximo y el valor mínimo en la calificación de Técnica ($T$) y se eliminan el valor máximo y el valor mínimo en la calificación de Presentación ($P$).
	$N' = N - 2$.
	15.3. Cálculo de Promedios y Puntaje Total
1. Promedio de Técnica: $\text{Prom}_{T} = (\sum T_j) / N'$.
2. Promedio de Presentación: $\text{Prom}_{P} = (\sum P_j) / N'$.
3. Puntaje Total: $\text{Puntaje Total} = \text{Prom}{T} + \text{Prom}{P}$. Este resultado debe mostrarse con dos decimales (#,##).
15.4. Manejo de Poomsaes Múltiples
Cuando la configuración requiere dos poomsaes, se califica el Poomsae 1 y luego el Poomsae 2. Posteriormente, los puntajes finales de ambos poomsaes (P1 y P2) se deben sumar y promediar para obtener el puntaje total del competidor en esa ronda o encuentro.
16. Algoritmo de Desempate (Lógica de Negocio 2)
La lógica de desempate se aplica cuando los $\text{Puntaje Total}$ de dos competidores son idénticos, o al definir el ranking en Rondas.
Paso
	Criterio de Comparación
	Acción Lógica
	Fuente
	Paso 1
	$\text{Puntaje Total}$ (Puntaje Total)
	Si son diferentes, gana el puntaje más alto.
	

	Paso 2
	$\text{Prom}_{P}$ (Presentación Promedio)
	Si $\text{Puntaje Total}$ es igual, se compara el promedio de Presentación. Gana el competidor con mayor $\text{Prom}_{P}$.
	

	Paso 3
	Empate Técnico
	Si el $\text{Prom}_{P}$ es igual, se declara Empate Técnico y el sistema debe habilitar el botón "Iniciar Poomsae de Desempate".
	

	Nota sobre Desempate Post-Rematch (Recalificación): Si se realiza un re-match (Poomsae adicional) y el puntaje resulta nuevamente en empate, el sistema debe recurrir al puntaje total original, incluyendo los puntajes máximos y mínimos que fueron descartados en el cálculo inicial. El competidor con el puntaje total más alto, utilizando todos los valores, será el ganador.
17. Lógica de Asignación de Puestos en Pirámides (Eliminación Directa)
Para las categorías de eliminación directa (1 vs 1), una vez definida la llave y jugados todos los encuentros hasta la Final, el sistema debe asignar los puestos 1°, 2°, 3° y 4° basándose en el flujo de la llave.
Puesto
	Definición
	Razón Lógica (Flujo del Torneo)
	Fuente
	1° Lugar
	Campeón
	Ganador del Encuentro Final.
	

	2° Lugar
	Subcampeón
	Perdedor del Encuentro Final.
	

	3° Lugar
	Bronce
	Se asigna al competidor que perdió en la Semifinal contra el Campeón.
	

	4° Lugar
	Bronce
	Se asigna al competidor que perdió en la Semifinal contra el Subcampeón.
	

	Nota técnica (VAI): El sistema debe gestionar encuentros donde uno o ambos competidores pasan por VAI (Bye). Si un competidor avanza por VAI (ausencia), el botón "Iniciar Puntuación" debe cambiar a "Guardar Ronda" para registrar el avance sin ingresar calificaciones, y el competidor avanza directamente a la siguiente fase de la pirámide.
